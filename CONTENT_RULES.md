# 內容規範（種子文件）

這不是完整的內容規範抽取（那個大工程還在 `ROADMAP.md` 的「之後可以考慮」清單裡），
是先把幾條已經定案、容易被未來的 AI 協作者忘記/犯錯的原則寫下來，避免每次重新踩雷。

---

## ✅ 新功能開工前的三個問題（2026-07-19 VERA定案，進入「穩定期」後的標準檢查）

專案已經建立五大支柱（內容／世界觀／導覽／維護工具／規範文件），之後任何新功能開工前，先問：

1. **有沒有重複建立資料？**（單一資料來源——見下方章節）
2. **是不是至少提供一個有效入口？**（不用追求所有入口都有——見下方章節）
3. **符合莊園世界觀嗎？**（命名/導覽/維護詞彙要一致，例如`maintenance.js`的健康度用🌱萌芽階段而不是工程百分比）

同時，**一個Sprint只做一個主題**，不要同一輪同時碰四五個系統——這條也是2026-07-19定案，跟上面三個問題一起用。

## 🔍 巡園→記錄→累積→一次收斂（2026-07-19 VERA定案，取代「全面盤查」）

2026-07-19這天密集做了好幾輪全站關鍵字盤查（文法術語/文字風格/字級），VERA最後定調：
**除非是真Bug，否則不要再做全面盤查（grep式掃描）**——因為剩下的問題大多不是關鍵字能找到的，
是「真正使用時才會浮現」的體感問題（例如「這句讀起來卡」「我不知道這按鈕幹嘛」）。

之後的工作模式改成：

1. VERA正常使用莊園，遇到卡頓/困惑/跟預期不同，就記一筆到 `UX_ISSUES.md`（不用停下來修、不用寫技術細節）
2. 累積5～10筆後，一次交給工程端批次處理
3. 工程端修完後把項目從「待處理」搬到「已收斂」，附commit/日期

`UX_ISSUES.md` 每筆固定四欄：📍位置／😕感覺／💡改善方向／✅狀態。這是之後判斷「要不要開工」的
主要依據，比起等VERA口頭描述或重新做一次全站掃描更有效率。**這條不影響「真Bug」的判斷**——
明確的功能壞掉（例如按鈕點了沒反應、資料顯示錯誤）還是直接修，不用等進UX_ISSUES.md排隊。

## 🌱 不為了填滿數字破壞品質（2026-07-19 VERA定案）

**背景**：`maintenance.js` 的「Chunk 家族健康檢查」會列出每個語塊家族分支在
劇情/歌曲/新聞/心語裡「有沒有出現過」。第一版報告把這些都當成「缺口」列出來，
容易讓人（包含未來的 AI 協作者）直覺想「趕快補滿」。

**VERA 的指正**：這些不是同一種東西，混在一起報會誤導：

1. **可補的（真缺口）**：維護者自己就能直接寫的內容——例如 `corazon.js`／`mom.js`
   的心語/日記短句，這些是原創內容，不涉及外部引用查證，發現缺口可以直接補。
2. **成長中的（不是缺口，是待成長區）**：受規則/架構限制，現在硬補只會犧牲品質：
   - 📺 **劇情**：受「每集固定10句」的 globalIdx 架構限制（`ep*10+idx`），
     不能幫舊集數塞新句子，只能等未來開新集數才能補這塊。
   - 🎵 **歌曲** / 📰 **新聞**：受「引用真實作品必須VERA親自查證」規則限制
     （見 `SONGS.md`／本文件件下方），AI 不能自己編一句歌詞或新聞標題湊數。

**規則**：任何診斷報告（`maintenance.js` 或未來類似工具）在列「缺口」時，
一律要區分這兩類，不能籠統列成同一種「缺口」。看到空白不代表要硬補——
先問「這是可以我自己直接寫的，還是需要等外部素材/新集數的？」。

## 🎵 外部引用（歌詞/新聞）查證鐵則

見 `SONGS.md`「收錄標準」與 `CLAUDE.md` 開發野史裡多次「隔壁AI編造引用被抓包」的教訓，摘要：

1. 任何歌詞/新聞的具體引用，AI 不能自己生成或憑記憶默寫，只能：
   - 用 WebSearch **列出真實搜尋結果**（標題+連結），不自己編寫內容
   - VERA 親自核對「三項規則」：①標題可搜尋到 ②原網址可開啟 ③內容與標題一致
2. 「查得到網址」不等於「內容真實」——網址真實、標題真實、內容摘要真實，三者要分開查證（曾抓到「真連結配假內容」的案例）。
3. 「隔壁AI」推薦的候選歌曲，看起來像傳統兒歌不代表真的是公共財——已有多次案例
   （Bartolito/Pin Pón/Hola Don Pepito/La vaca Lola等）其實是有名有姓的現代版權作品，
   一律要先查證作曲人/版權狀態再收錄。

## 🌳 每個語塊至少要有一個入口，而不是每個入口都要有（2026-07-19 VERA定案）

一個語塊理論上可以從六個地方被使用者遇到：🌾Grammar／📺劇情／🎵歌曲／📔心語／📰新聞／🪞腦中劇場。

**核心原則**：不要求全部都有，只要求「至少有一個能讓使用者接觸它」。找不到歌不代表這個語塊沒有入口，
只是暫時少一種入口——`maintenance.js`的「Chunk家族健康檢查」用「至少有一個入口的分支數」當真正的
健康指標，而不是逐項比對各來源的滿足率。逐項滿足率（📺/🎵/📰各自的比例）只是參考，不是KPI，
不要為了填滿某一項的數字去犧牲內容品質（呼應本文件最上面那條原則）。

## 🔗 單一資料來源（Single Source of Truth，2026-07-19 VERA定案）

同一份內容只存一份，其他地方一律用「引用」不要「複製」。例如：
- 心語只有一份（`corazon.js`／`mom.js`），🌳語塊茁壯的📔徽章跟🛌床邊低語呢共用同一份資料，
  不是各自維護一份心語清單
- 文法只有一份（`grammar.js`），劇情索引/語塊家族/儲水槽都是「連結到」同一張卡，不是複製內容進去
- 歌曲只有一份（`LYRICS_FILL_DATA`），語塊家族的🎵徽章、每日任務的歌曲入口，都是連到同一筆資料

**之後任何新功能需要用到已有的內容類型時，先問「這個資料已經存在了嗎？」，能複用就複用，
不要因為想做新功能就開一份新的平行資料。**

## 🌿 成長資料分層原則（2026-07-19定案：大方向先定，畫面不用先蓋）

VERA指正：🌻語塊花園跟🎖️蠟封晉級證書回答的是不同問題，不應該互相取代——但「我的成長」這個名字
本身天然包含三層，這個資訊架構現在就可以先定下來，之後不管誰接手都不會誤用：

```
🌿 我的成長 ＝ 我的西語生命紀錄（總入口概念，目前尚未做成畫面）
├─ 🌱 語塊成長：現在哪些語塊正在長大 → 🌻語塊花園（逐株照顧／植物狀態）
├─ 🎖️ 成長足跡：曾經完成哪些階段 → 🎖️蠟封晉級證書（人生履歷／莊園歷史）
└─ 🕰️ 成長時間：何時開始、累積多久 → getFirstChunkDate()/daysSinceFirstChunk()（日期資料）
```

**規則**：這三層資料各自獨立存在，不要把徽章塞進語塊花園的資料裡，也不要因為要顯示日期就
另開一個日期系統——`getFirstChunkDate()`已經是這三層共用的時間軸，直接引用即可。

**現在只定「資料邏輯」跟「名稱定位」，UX呈現方式（要不要做總覽頁/哪個排第一/卡片長什麼樣/
要不要合併顯示）留到真人巡園觀察使用者實際點進「我的成長」期待看到什麼之後再決定**——避免先蓋一個
空殼總覽頁，結果發現使用者其實只想看「我今天該做什麼」，白白浪費一次重構。目前header的
🌿我的成長維持連去🌻語塊花園（見上方），這個決定本身沒有錯，只是還沒到「總覽頁」那一步。

## 💎🌻 私語窖／花園 資料責任分離（2026-07-19定案：分層合理，不要合併）

VERA確認💎醞釀私語窖跟🌻語塊花園分開存在是健康的設計，不是需要合併的重複系統——兩者回答完全
不同的問題，且已經是「單一資料來源＋互相引用」（私語窖卡片的熟練度圖示是即時去查花園那份資料，
私語窖自己不存熟練度，見上面「兩套系統跟收容的差異」的問答）：

| 系統 | 管什麼 | 不管什麼 | localStorage key |
| --- | --- | --- | --- |
| 🌻 語塊花園 | 熟練度、生長階段、累積歷程（全站任何地方點星星都算） | 使用者是否收藏過 | `peppa_garden_v1` |
| 💎 私語窖 | 收藏意願、來源位置（哪個分區收藏來的）、私人書籤 | 熟練度（顯示的星星是引用花園資料，自己不存） | `peppa_es_vocab_v1` |

三者可以同時發生、互不取代：💎收藏＝「我想留意」／🌻種進花園＝「我要培養」（點vocab卡片右側⭐或
任何地方的語塊星星，兩者共用同一個`handleGardenProgress()`）／🐛抓蟲複習＝「需要回來照顧」。

**刪除行為規則**：
- 從💎私語窖移除書籤（`removeFromVocab()`）→ 只刪vocabList，**不影響**🌻花園熟練度（現況已經
  是這樣，`removeFromVocab()`沒有碰`peppa_garden_v1`，行為正確，之後不要加這個副作用進去）
- 🌻花園目前沒有「主動刪除單一語塊」的功能（熟練度只會往前進，沒有刪除入口），這條規則暫時
  用不到，但如果之後真的要做花園刪除功能，要同步確認：私語窖裡對應的書籤是否還保留？要不要
  顯示「來源已下架」？——現在先記錄這個備忘，不用因為現在沒有這個功能就不管。

**使用者心智模型缺口**（2026-07-19盤點，不是功能缺口，是導覽缺口）：VERA自己實測後發現「收藏了
為什麼沒有出現在抓蟲複習」——因為💎收藏（點＋）純粹是書籤動作，不會自動觸發🌻花園追蹤，要另外
點🌱／⭐星星才會真正開始被複習系統看見。這代表問題不在架構（架構是對的），在於**沒有一張說明
「莊園裡三種紀錄分別是什麼」的導覽牌**。已在💎私語窖小標題加上一句提示（「熟練度⭐來自🌻語塊花園，
收藏本身不會自動開始追蹤」），**還沒做**：🗺️莊園導覽補一頁完整說明「💎我想留 / 🌻我正在學 /
🐛我需要補強」三句話總結，等莊園巡園週的其他真人測試項目一起排。

## 🏷️ 文法卡提示標籤系統（2026-07-19定案，之後新增提示不要自己發明樣式）

`openGrammarCard()`裡的補充說明欄位，統一用固定的「badge＋box」樣式（`.grammar-tag-box`／
`.grammar-tag-badge`／`.grammar-tag-body`，見script.js/style.css），身份靠badge文字＋顏色識別，
不是靠字體大小。目前有兩種在用：

- **⚠️ 常見陷阱**（`g.trap`欄位，紅色）：容易犯錯，但不一定違反直覺——例如固定搭配、常見誤用
- **🐛 調皮特例**（`g.quirk`欄位，紫色虛線框，比常見陷阱更醒目）：打破初學者的直覺推論，例如
  「明明看起來像A邏輯，其實用B」這種會讓人形成錯誤心智模型的例外規則

寫`quirk`內容時，**先講規則再講例子**，不要只丟一句例句——格式建議：
```
❌ 直覺：（初學者直覺會怎麼想）
✅ 其實是：（真正的規則/邏輯）
例句對照（❌錯誤講法 → ✅正確講法，或用≠對比兩個不同情境）
```
已套用範例：g01（SER的婚禮地點）、g02（ESTAR的死亡狀態）、g109（hacer frío天氣現象）。

**之後如果真的需要「🌱核心規則卡」（一定要掌握的公式）或「🔗對比橋」（中文思維→西語思維的轉換），
比照這個結構加一組`.tag-核心規則`/`.tag-對比橋`即可，不要另外設計一套新樣式。**

## 📏 手機閱讀字級下限（2026-07-19 VERA定案，舊卡不回頭改，新卡一律套用）

盤查發現全站 style.css 有 100+ 個 class 的 `font-size` 低於 12px（9px～11.5px），分佈在幾乎每個
分頁的badge/小標籤/註解文字上，是長期累積的既有設計慣例，**不做一次性全站回頭放大**（範圍太大，
會動到幾乎每個卡片的版面，風險不成比例）。改採「舊的不動、新的守規則」：之後新增任何卡片/語塊/
文法區塊的文字，一律照這個字級表：

- 🌱 主標題：18px
- 🧩 西語核心句：16px
- 中文解釋：15–16px
- 補充提示／小標籤：14px以上
- **最小不得低於12px**（標點符號本身可例外縮小，但不能是唯一理由把整段文字都壓小）

檢查時不要只看CSS數字，要在390px寬手機視角實際看：中文是否需要瞇眼、長句是否擠在一起、行高是否足夠。
現有最小的幾處供之後參考（非優先修復名單，只是紀錄基準）：`.ge-chunk-star`(8px)、
`.grammar-tip-label`/`.vocab-source`/`.grammar-source`/`.conj-tense-hint`/`.garden-chip-qc`(9px)。

## 💬 動詞變位/文法提示：只用「意思優先」，不用課本術語標籤（2026-07-19 VERA定案）

補充說明動詞變化時，**不要**用「第一人稱單數」「第三人稱複數」這類純文法術語當作解釋本身——
這是傳統課本式標籤，會破壞莊園「白話理解、從句子長出規律」的原則。已盤查全站，`buildConjTable()`/
`renderConjLibrary()`（🔄超級變變變）與 ammo.js 的 chunk `note` 欄位本來就是「意思優先」寫法
（例：`ser ➔ soy（「我」）➔ 意思：我是`），不需要調整；但 news.js（📰B2時事傳送門）53則填空題裡，
有相當多筆 `hint` 欄位是純文法術語（`verbo, 第三人稱單數`／`sustantivo f. pl.`／`adjetivo m.`），
沒有先講意思再補規律，之後改寫這些hint時比照下面格式：

```
不要：hint: 'verbo, 第三人稱單數 — poner en peligro algo'
改成：hint: '💡 amenaza = 正在威脅／構成威脅
      來自 amenazar（威脅）。主詞是「它/這件事」時，動詞會變成 amenaza。'
```

## 🌳 架構限制備忘

- **episodes.js**：`globalIdx = ep*10+idx` 全站共用這個固定跨距（`total()`／`showGrammarTip()`／
  `accumulateSVOPool()`／`currentGlobalIdx()`多處依賴），代表**每集必須剛好10句**，
  不能多也不能少，想擴充內容只能開新集數，不能塞進舊集數裡。
